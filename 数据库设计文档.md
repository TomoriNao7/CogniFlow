# CogniFlow 数据库设计文档

## 1. 设计原则

基于数据库三大范式进行规范化设计，目标：减少数据冗余、避免更新异常、保证数据一致性。

| 范式 | 核心要求 | 解决什么问题 |
|------|----------|------|
| 第一范式（1NF） | 每个字段原子化，无重复组 | 避免一个字段塞多个值，导致查询和更新困难 |
| 第二范式（2NF） | 非主键字段完全依赖整个主键 | 避免联合主键场景下的部分依赖 |
| 第三范式（3NF） | 非主键字段不依赖于其他非主键字段 | 消除传递依赖，避免数据不一致 |

规范化的代价是表变多、JOIN 变多。对于高频读取场景，可用视图或物化视图做反范式优化，但基础存储必须规范。

---

## 2. 字段规范

### 2.1 软删除

所有业务实体表统一加 `deleted_at` 字段，删除操作不物理删除数据，仅标记时间戳。查询时通过 `WHERE deleted_at IS NULL` 过滤有效数据。

**需要软删除的表**：users、admin_users、conversations、agents、tools、knowledge_documents、human_handoffs、feedbacks

**不需要软删除的表**（依赖父表存在，父表软删除后自然不可见）：
user_attributes、user_tags、messages、message_traces、trace_retrieval_chunks、trace_function_calls、agent_retrieval_params、tool_parameters、agent_tools、knowledge_chunks、chunk_metadata、chunk_tags、handoff_slots、handoff_agent_attempts

每个软删除表配合一个部分索引，加速有效数据查询：

```sql
CREATE INDEX idx_xxx_active ON xxx(deleted_at) WHERE deleted_at IS NULL;
```

### 2.2 密码存储

所有密码字段仅存储**哈希值**，绝不存明文。使用 bcrypt（cost ≥ 12）或 argon2id 算法。字段命名为 `password_hash` 明确标识不可逆。

### 2.3 金额与货币

涉及金额的字段一律**不存浮点数**（FLOAT/DOUBLE 有精度丢失风险），选用以下两种方式之一：

| 方式 | 类型 | 示例 | 适用场景 |
|------|------|------|----------|
| 整数（分） | `INTEGER` / `BIGINT` | 1999 表示 19.99 元 | 计算密集、需高性能 |
| 精确小数 | `NUMERIC(12,2)` | 19.99 | 需直接可读、存储为主 |

货币类型统一为 **CNY**（人民币），如后续需要多币种，可在金额所在表的相邻字段加 `currency VARCHAR(3) DEFAULT 'CNY'`。

**本设计中涉及金额的字段均使用 `NUMERIC(12,2)`，最大支持 9,999,999,999.99 元。**

## 3. 违反点识别与修正

### 3.1 第一范式（1NF）：拆解非原子字段

| 表 | 字段 | 违规原因 | 修正方式 |
|------|------|------|------|
| users | profile (JSONB) | 内含标签列表、偏好键值对，实为重复组 | 拆为 user_attributes + user_tags |
| tools | parameter_schema (JSONB) | 内含多条参数定义，每条参数是独立实体 | 拆为 tool_parameters |
| knowledge_chunks | metadata (JSONB) | 内含标题层级、标签数组，非原子值 | 拆为 chunk_metadata + chunk_tags |
| messages | trace_data (JSONB) | 内含分类/检索/重排/工具/生成五步的嵌套数据 | 拆为 message_traces + trace_retrieval_chunks + trace_function_calls |
| human_handoffs | context_snapshot (JSONB) | 内含摘要、槽位、尝试记录的嵌套结构 | 拆为 human_handoffs 字段 + handoff_slots + handoff_agent_attempts |
| agents | retrieval_config (JSONB) | 内含多个独立配置项 | 拆为 agent_retrieval_params |

### 3.2 第二范式（2NF）

本设计以自增 ID 为主键，不存在联合主键，2NF 自然满足。唯一关注 `agent_tools` 表，其逻辑联合键为 (agent_id, tool_id)，tool_version 同时依赖于两者，符合 2NF。

### 3.3 第三范式（3NF）：消除传递依赖与派生字段

| 表 | 字段 | 传递依赖链 | 修正方式 |
|------|------|------|------|
| feedbacks | conversation_id | feedbacks → message_id → conversation_id | 删除，JOIN messages 获取 |
| knowledge_documents | total_retrievals | 可由 SUM(chunks.retrieval_count) 派生 | 删除，用视图替代 |
| knowledge_documents | chunk_count | 可由 COUNT(chunks) 派生 | 删除，用视图替代 |
| conversations | message_count | 可由 COUNT(messages) 派生 | 删除 |
| conversations | satisfaction | 可由 feedbacks 汇总派生 | 删除，用视图替代 |
| messages | feedback / feedback_reason | 与 feedbacks 表职责重叠 | 删除 |

## 4. 完整表结构

---

### 4.1 用户表 `users`

终端消费者，通过电商平台用户 ID 关联。支持软删除。

```sql
CREATE TABLE users (
    id            SERIAL PRIMARY KEY,
    external_id   VARCHAR(128) NOT NULL UNIQUE,
    nickname      VARCHAR(128),
    phone         VARCHAR(256)   NOT NULL,       -- AES-256 加密存储
    email         VARCHAR(256),                   -- AES-256 加密存储
    created_at    TIMESTAMP DEFAULT NOW(),
    updated_at    TIMESTAMP DEFAULT NOW(),
    deleted_at    TIMESTAMP                       -- 软删除标记
);

CREATE INDEX idx_users_external_id ON users(external_id);
CREATE INDEX idx_users_active      ON users(deleted_at) WHERE deleted_at IS NULL;
```

**1NF 修正**：profile JSONB 拆为 user_attributes 和 user_tags。
**字段规范**：deleted_at 实现软删除，phone/email 加密存储。

---

### 4.2 用户属性表 `user_attributes`

原 profile JSONB 中的键值对，每行一个属性，满足 1NF 原子性。父表软删除后通过 JOIN 自然不可见，无需独立 deleted_at。

```sql
CREATE TABLE user_attributes (
    id          SERIAL PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id),
    attr_key    VARCHAR(64) NOT NULL,            -- membership / preferred_category
    attr_value  VARCHAR(256) NOT NULL,           -- gold / 电子产品
    created_at  TIMESTAMP DEFAULT NOW(),

    UNIQUE (user_id, attr_key)
);

CREATE INDEX idx_ua_user_id ON user_attributes(user_id);
```

**设计理由**：属性随用户终身存在，按 user_id 查。长期记忆注入 Agent prompt 时从这里读取。

---

### 4.3 用户标签表 `user_tags`

原 profile JSONB 中 tags 数组，每个标签一行，满足 1NF 消除重复组。

```sql
CREATE TABLE user_tags (
    id          SERIAL PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(id),
    tag         VARCHAR(64) NOT NULL,            -- 退款敏感 / 高频投诉 / VIP
    created_at  TIMESTAMP DEFAULT NOW(),

    UNIQUE (user_id, tag)
);

CREATE INDEX idx_ut_user_id ON user_tags(user_id);
```

**设计理由**：标签可动态增减，UNIQUE 约束防止重复打标。

---

### 4.4 管理员表 `admin_users`

后台管理系统登录用，与终端消费者物理隔离。支持软删除。

```sql
CREATE TABLE admin_users (
    id            SERIAL PRIMARY KEY,
    username      VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(256) NOT NULL,          -- bcrypt(cost=12) 哈希，不可逆
    role          VARCHAR(32) DEFAULT 'operator', -- admin / operator / viewer
    status        VARCHAR(16) DEFAULT 'active',
    created_at    TIMESTAMP DEFAULT NOW(),
    updated_at    TIMESTAMP DEFAULT NOW(),
    deleted_at    TIMESTAMP
);

CREATE INDEX idx_au_active ON admin_users(deleted_at) WHERE deleted_at IS NULL;
```

**安全说明**：password_hash 使用 bcrypt（cost ≥ 12），绝不明文存储。即使数据库泄露，密码不可逆。

---

### 4.5 会话表 `conversations`

删除 message_count 和 satisfaction，消除 3NF 派生字段冗余。支持软删除。

**金额字段说明**：本表不直接存储金额，但会话关联的订单金额由外部业务系统管理。如需缓存订单金额供 Agent 快速展示，使用 `NUMERIC(12,2)`，参见 2.3 字段规范。

```sql
CREATE TABLE conversations (
    id            SERIAL PRIMARY KEY,
    user_id       INTEGER NOT NULL REFERENCES users(id),
    agent_id      INTEGER REFERENCES agents(id),
    status        VARCHAR(16) DEFAULT 'active',  -- active / closed / handoff
    title         VARCHAR(256),
    created_at    TIMESTAMP DEFAULT NOW(),
    updated_at    TIMESTAMP DEFAULT NOW(),
    deleted_at    TIMESTAMP
);

CREATE INDEX idx_conv_user_id    ON conversations(user_id);
CREATE INDEX idx_conv_status     ON conversations(status);
CREATE INDEX idx_conv_agent_id   ON conversations(agent_id);
CREATE INDEX idx_conv_created_at ON conversations(created_at);
CREATE INDEX idx_conv_active     ON conversations(deleted_at) WHERE deleted_at IS NULL;
```

**3NF 修正**：
- `message_count` → 用 `COUNT(*) FROM messages WHERE conversation_id = ?`
- `satisfaction` → 用 conversation_satisfaction 视图

---

### 4.6 消息表 `messages`

不独立加 deleted_at。软删除会话时，通过 JOIN conversations 过滤。

```sql
CREATE TABLE messages (
    id              SERIAL PRIMARY KEY,
    conversation_id INTEGER NOT NULL REFERENCES conversations(id),
    role            VARCHAR(8) NOT NULL,        -- user / bot / system
    content         TEXT,
    created_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_msg_conv_id    ON messages(conversation_id);
CREATE INDEX idx_msg_created_at ON messages(conversation_id, created_at);
```

**1NF + 3NF 修正**：
- `trace_data` → 拆为 message_traces + trace_retrieval_chunks + trace_function_calls
- `feedback` / `feedback_reason` → 统一由 feedbacks 表管理

---

### 4.7 消息追踪表 `message_traces`

原 trace_data JSONB 拆解。一次 bot 回复对应一条记录，每列存原子值，满足 1NF。

```sql
CREATE TABLE message_traces (
    id                          SERIAL PRIMARY KEY,
    message_id                  INTEGER NOT NULL UNIQUE REFERENCES messages(id),

    -- 意图分类
    intent_agent                VARCHAR(64),
    intent_confidence           DECIMAL(4,3),
    intent_duration_ms          INTEGER,

    -- 检索
    retrieval_query             TEXT,
    retrieval_recalled_count    INTEGER,
    retrieval_top_k             INTEGER,
    retrieval_duration_ms       INTEGER,

    -- 重排序
    rerank_model                VARCHAR(64),
    rerank_duration_ms          INTEGER,

    -- 生成
    generation_model            VARCHAR(32),
    generation_prompt_tokens    INTEGER,
    generation_completion_tokens INTEGER,
    generation_ttft_ms          INTEGER,
    generation_duration_ms      INTEGER,

    created_at                  TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_mt_message_id ON message_traces(message_id);
```

**设计理由**：每一步的耗时独立为列，可直接做 P50/P99 延迟分析和 Agent 性能对比。仅 bot 消息有此记录。

---

### 4.8 检索结果明细表 `trace_retrieval_chunks`

原 trace_data.retrieval.chunks 数组的规范形式。每个召回的切片一行。

```sql
CREATE TABLE trace_retrieval_chunks (
    id              SERIAL PRIMARY KEY,
    trace_id        INTEGER NOT NULL REFERENCES message_traces(id),
    document_id     INTEGER NOT NULL REFERENCES knowledge_documents(id),
    chunk_id        INTEGER NOT NULL REFERENCES knowledge_chunks(id),
    score           DECIMAL(6,4) NOT NULL,
    rank_position   INTEGER NOT NULL
);

CREATE INDEX idx_trc_trace_id ON trace_retrieval_chunks(trace_id);
```

**评估用途**：可对单次检索计算召回率和精确率，按时间聚合评估检索质量趋势。

---

### 4.9 工具调用明细表 `trace_function_calls`

原 trace_data.function_calling 的规范形式。一次消息可能多次调工具（含重试），每次一行。

```sql
CREATE TABLE trace_function_calls (
    id              SERIAL PRIMARY KEY,
    trace_id        INTEGER NOT NULL REFERENCES message_traces(id),
    tool_name       VARCHAR(64) NOT NULL,
    tool_version    VARCHAR(16),
    parameters      TEXT,                        -- 调用的入参，快照记录
    is_success      BOOLEAN NOT NULL,
    result_summary  TEXT,
    attempt_index   INTEGER DEFAULT 0,           -- 0=首次
    duration_ms     INTEGER,
    created_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_tfc_trace_id ON trace_function_calls(trace_id);
```

**设计理由**：parameters 字段存序列化文本用于问题复现，不拆表——参数结构随工具类型变化，按参数名查询需求极低。

### 4.13 工具参数表 `tool_parameters`

原 parameter_schema JSONB 中每个参数定义拆为一行，满足 1NF。

```sql
CREATE TABLE tool_parameters (
    id              SERIAL PRIMARY KEY,
    tool_id         INTEGER NOT NULL REFERENCES tools(id),
    param_name      VARCHAR(64) NOT NULL,
    param_type      VARCHAR(32) NOT NULL,        -- string / integer / boolean / object / array / numeric

CREATE INDEX idx_tp_tool_id ON tool_parameters(tool_id);
```

**字段规范说明**：param_type 支持 `numeric` 类型，用于标识金额参数。金额类型参数传入时以字符串形式传递（如 `"19.99"`），后端转为 `NUMERIC(12,2)` 处理。

```sql
CREATE TABLE agent_tools (
```

### 4.16 知识文档统计视图

```sql
CREATE VIEW knowledge_document_stats AS
SELECT
    d.id,
    d.title,
    COUNT(c.id)                              AS chunk_count,
    COALESCE(SUM(c.retrieval_count), 0)      AS total_retrievals
FROM knowledge_documents d
LEFT JOIN knowledge_chunks c ON c.document_id = d.id
WHERE d.deleted_at IS NULL
GROUP BY d.id, d.title;
```

**注意**：视图已过滤 deleted_at IS NULL，后台查询直接使用视图即可。

### 4.21 会话满意度视图

```sql
CREATE VIEW conversation_satisfaction AS
SELECT
    c.id AS conversation_id,
    f.rating,
    f.reason,
    f.created_at
FROM conversations c
JOIN messages m ON m.conversation_id = c.id
JOIN feedbacks f ON f.message_id = m.id
WHERE c.deleted_at IS NULL
  AND f.deleted_at IS NULL;
```

### 8.6 关键关系补充说明

**Milvus 与 PostgreSQL 的映射**：知识切片存入 PostgreSQL（内容+元信息），同时写入 Milvus（chunk_id + 向量）。检索流程为：

```
用户问题 → 向量化 → Milvus 检索 → 返回 chunk_id 列表 → PostgreSQL 批量取 content
```

两张表中 chunk_id 一一对应，Milvus 不存内容，只存向量和 chunk_id。

**feedbacks 与 messages 的基数**：一个 message 可以有多条 feedback（用户可能修改反馈），因此 1:N 关系，不加 UNIQUE(message_id) 约束，保留历史。
