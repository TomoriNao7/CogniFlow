# CogniFlow 电商智能客服 — 立项文档

## 1. 项目概述

CogniFlow 是一款面向电商场景的企业级智能客服系统，采用多 Agent 协作架构，为电商平台提供售前、售中、售后全链路的智能对话服务。

### 1.1 核心目标

- 基于多领域 Agent 协作，覆盖电商售前、售中、售后三大场景
- 混合检索 + 重排序的 RAG 方案，支持多格式知识库
- Function Calling 驱动业务工具调用，实现"能说也能做"
- 完整的反馈闭环，持续优化回答质量和知识覆盖

---

## 2. 前端界面

系统包含两套独立的前端应用，面向不同的用户角色。

### 2.1 用户端 Widget（面向终端消费者）

嵌入电商官网的聊天窗口，消费者直接使用。

| 项目 | 选型 |
|------|------|
| 形态 | 网页 Widget（嵌入官网） |
| 使用者 | 终端消费者 |
| 技术栈 | React + WebSocket |
| 通信方式 | WebSocket 长连接，支持流式输出 |
| 核心功能 | 聊天窗口、消息气泡、富文本渲染、反馈按钮 |

### 2.2 后台管理系统（面向运营 / 客服主管）

独立的 Web 应用，管理知识库、工具、Agent、对话记录。

| 项目 | 选型 |
|------|------|
| 形态 | 独立 Web 应用 |
| 使用者 | 运营人员、客服主管 |
| 技术栈 | React（可复用组件生态） |
| 通信方式 | HTTP REST API（FastAPI 提供） |

**后台管理功能模块**：

| 模块 | 功能 |
|------|------|
| 知识库管理 | 文档上传、格式自动识别、切片预览、版本管理、检索统计 |
| 工具注册 | 可视化注册工具、绑定 Agent、在线测试、启用/禁用 |
| Agent 配置 | 绑定模型、调整 system prompt、调整检索参数、查看指标 |
| 对话记录 | 筛选、回放、决策追溯、人工标注 |
| 反馈看板 | 知识盲区 Top 10、高转人工率排行、Agent 表现对比 |

### 2.3 两套前端对比

| | 用户端 Widget | 后台管理系统 |
|------|------|------|
| 使用者 | 终端消费者 | 运营 / 主管 |
| 形态 | 嵌入式聊天窗口 | 独立 Web 应用 |
| 目标 | 获取答案、办理业务 | 配置知识库、监控效果 |
| 通信 | WebSocket 长连接 | HTTP REST |
| 鉴权 | 无（或跟随官网登录态） | 账号密码登录 |

### 2.4 接入层

两套前端共用同一套接入层：

| 组件 | 职责 |
|------|------|
| CDN / Nginx | 两套前端的静态资源分发 |
| API Gateway | 鉴权、限流、请求路由 |
| WebSocket 服务 | 仅用户端 Widget 连接，长连接管理，流式推送 |

## 3. 核心架构

### 3.1 总览

```
用户问题
  → 内容安全检测
  → 意图分类（Qwen 小模型）
  → 路由到领域 Agent
    → LangGraph 编排
    → 短期记忆（摘要压缩）
    → 混合检索 + bge-reranker 重排序
    → Function Calling 工具调用
    → 分模型生成回复（Max / Plus / Turbo）
  → 失败兜底 / 高风险 → 转人工
```

### 3.2 多 Agent 分工

采用按领域划分 Agent 的方式，由意图分类模型路由分发。

| Agent | 负责场景 | 知识范围 | 工具 |
|-------|----------|----------|------|
| 售前 Agent | 商品咨询、规格对比、库存、优惠、会员 | 商品库、促销规则、会员政策 | 查商品、查库存、查优惠券、查会员等级 |
| 售中 Agent | 下单问题、支付失败、地址修改、发票 | 支付说明、发票规则、物流政策 | 查未支付订单、修改地址、申请发票 |
| 售后 Agent | 查物流、退款/退货、投诉、质量问题 | 退换货政策、售后流程、投诉规则 | 查物流、查订单、发起退款、查退款进度 |

路由逻辑：
- "这个商品有货吗" → 售前
- "我付了钱怎么没成功" → 售中
- "货到哪了"、"我要退款" → 售后
- 边界模糊问题 → 售后优先

### 3.3 领域 Agent 内部

每个领域 Agent 内部采用单 Agent + RAG + 工具的模式，不拆分子 Agent。流程由 LangGraph 编排。

## 4. 模型选型

统一使用 Qwen 系列云端模型，按场景分模型以控制成本。

| 用途 | 模型 | 说明 |
|------|------|------|
| 意图分类 | Qwen 小模型（如 0.5B/1.5B） | 轻量分类，低延迟 |
| 售前生成 | Qwen Max | 商品对比、推荐类问题复杂度高 |
| 售中生成 | Qwen Plus | 中等复杂度，性价比优先 |
| 售后生成 | Qwen Max | 投诉、退款类要求高准确率 |

## 5. RAG 知识库

### 5.1 多格式分切策略

不同文档格式走不同切分管线：

| 文档格式 | 切分方式 |
|----------|----------|
| Markdown（帮助文档） | 按文档结构切（H1/H2/H3） |
| FAQ/QA 对 | 按 Q&A 自然边界切 |
| PDF/Word（白皮书、规范） | 语义切片 |
| 纯文本（公告、通知） | 固定大小切片 + 滑动窗口 |
| 表格/Excel | 按行或按表切，保留表头 |

上传文档时自动识别格式，走对应管线。

### 5.2 检索策略

- **混合检索**：BM25 关键词 + 向量语义检索，多路召回
- **重排序**：bge-reranker-v2-m3 本地部署，对召回结果精排
- **目标**：电商 FAQ 场景保精确率，长尾问题保召回率

### 5.3 文档管理

- 文档版本管理（更新不覆盖，可回滚）
- 每篇文档展示检索命中次数、被提问次数

---

## 6. 对话管理

### 6.1 记忆系统

| 类型 | 策略 | 存储 |
|------|------|------|
| 短期记忆 | 当前会话上下文，超过 10 轮做摘要压缩 | 内存（注入 prompt） |
| 长期记忆 | 用户画像、历史问题、偏好 | PostgreSQL |
| 工作记忆 | Agent 执行中间结果（查询数据、工具返回） | 会话周期内，不持久化 |

### 6.2 Function Calling

采用模型原生 Function Calling，Agent 自主判断是否调工具、调哪个、传什么参。

### 6.3 工具调用容错

| 机制 | 策略 |
|------|------|
| 重试 | 固定次数重试，递增间隔（1s → 2s → 4s） |
| 错误区分 | 网络超时可重试，业务错误不重试 |
| 熔断 | 工具连续失败 N 次，熔断一段时间 |
| 用户提示 | 失败返回有用信息，建议稍后重试或转人工 |

### 6.4 工具注册管理

集中注册表统一管理所有工具：
- 工具名、描述、参数 schema
- 归属领域 Agent
- 调用地址、超时时间、重试策略
- 版本号、启用/禁用

Agent 启动时从注册表拉取本领域工具列表，注入 system prompt。

## 7. 安全与合规

### 7.1 内容安全分级

| 级别 | 行为 | 处理 |
|------|------|------|
| 轻度（负面情绪） | 正常处理 + 安抚策略 | 不拦截 |
| 中度（侮辱性、恶意调侃） | 拦截，中性提示 | "请重新描述您的问题" |
| 严重（越狱、注入、恶意链接） | 静默拦截 + 告警记录 | "抱歉，我无法处理这个请求" |
| 连续违规 | 3 次 → 转人工 | 坐席判断 |

核心原则：不向用户暴露检测机制，表现为能力边界而非态度问题。

### 7.2 数据安全

- 对话数据存储于自建 PostgreSQL
- 敏感信息（身份证、银行卡、密码）检测并脱敏
- 生成内容合规审核（大促价格、优惠券描述等）

## 8. 人工兜底

### 8.1 转人工触发条件（优先级从高到低）

1. 高风险操作（退款、投诉、账号注销）
2. 用户主动要求转人工
3. 情感检测负面情绪累积
4. Agent 连续多次无法回答

### 8.2 转人工上下文传递

转人工时将以下信息打包传递：
- 对话摘要
- 已收集的槽位信息
- Agent 尝试过的工具和结果
- 用户意图分类结果

### 8.3 排队体验

- 排队期间展示预估等待时间
- 用户可选"排队中继续问 Agent"

---

## 9. 反馈闭环

### 9.1 对话层埋点

- 每轮对话命中 Agent、检索文档、工具耗时、生成耗时
- 回答采纳率（用户问完即停 vs 继续追问）

### 9.2 知识库健康度

- 未检索知识标记（冗余内容）
- 低相关性检索标记（知识缺失或文档不佳）
- 转人工后坐席解决的问题 → 自动标记为知识盲区

### 9.3 用户主动反馈

- 每个回答底部"有帮助 / 没帮助"按钮
- "没帮助"时弹可选追问"哪里不好？"
- 低赞问题自动进待优化队列

### 9.4 人工复核

- 转人工对话由坐席标注 Agent 失败原因（理解错误 / 知识缺失 / 工具失败）
- 标注结果自动汇总，形成优化待办清单

### 9.5 闭环流程

```
用户反馈 + 坐席标记 → 找短板 → 补知识 / 修工具 / 调 Agent → 指标改善
```

## 10. 评估体系

### 10.1 离线评估（上线前）

| 评估项 | 方法 | 指标 |
|--------|------|------|
| 意图分类 | 标注测试集 | 准确率 |
| 检索质量 | 给定问题，评估召回文档 | 召回率、精确率、命中率（Hit Rate）、MRR |
| 工具调用 | 给定对话，评估调用正确性 | 工具调用准确率 |
| 生成质量 | 模型或人工评分 | 准确率、完整率、幻觉率 |

### 10.2 在线评估（上线后）

| 维度 | 指标 | 目标 |
|------|------|------|
| 响应时间 | 首字延迟（TTFT） | < 2s |
| 响应时间 | 完整回复时间 | < 8s |
| 响应时间 | 端到端延迟 | < 10s |
| 用户行为 | 问了就走率、追问率、转人工率 | 趋势监控 |
| 满意度 | 点赞率 | 按 Agent 统计 |
| 工具 | Function Calling 成功率 | 按工具统计 |

### 10.3 A/B 对比

- 新版本上线与旧版本对比所有核心指标
- 大促和平峰数据分开展示

## 13. MVP 范围

### 13.1 MVP 包含

- [ ] React Widget 前端（WebSocket 长连接）
- [ ] 意图分类 → 售前 / 售中 / 售后三 Agent
- [ ] 知识库：Markdown + FAQ 两种格式
- [ ] 混合检索 + bge-reranker 重排序
- [ ] 至少一个工具（查订单），跑通 Function Calling 链路
- [ ] 短期记忆 + 摘要压缩
- [ ] 3 次失败转人工
- [ ] 基础反馈按钮（有帮助 / 没帮助）

### 13.2 MVP 暂不包含

- PDF / Word / Excel 等其他格式切分
- 大促弹性伸缩
- 情感检测
- 熔断降级
- 完整后台管理系统
- 高级评估体系

### 13.3 MVP 目标

跑通全链路：Widget 发消息 → 分类路由 → Agent 检索知识 → 调工具 → 生成回复 → 成功返回。

## 14. 部署架构

```
用户浏览器
  → CDN / Nginx（静态资源）
  → API Gateway（鉴权、限流）
  → WebSocket 服务（长连接、流式输出）
  → 后端服务（FastAPI）
    → 对话编排（LangGraph）
    → Agent 管理（售前/售中/售后独立服务）
    → 知识库管理
  → 依赖服务
    → PostgreSQL（会话、用户、工具注册表）
    → Redis（热会话缓存、热点问题缓存）
    → Milvus（向量检索）
    → 对象存储（知识库原始文件）
```

各领域 Agent 独立部署为服务实例，互不影响。WebSocket 支持水平扩展，通过 Redis Pub/Sub 做跨实例消息路由。
